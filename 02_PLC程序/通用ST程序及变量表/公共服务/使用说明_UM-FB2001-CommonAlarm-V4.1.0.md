## 2. 前置条件与依赖项

### 2.1 硬件环境要求

| 项目 | 要求 | 说明 |
|------|:----:|:-----|
| **PLC型号** | 支持ST语言的控制器 | 必须支持结构化文本编程和ARRAY/WORD数据类型 |
| **D区寄存器** | D400~D425可用 | 用于HMI显示和MES数据读取(共26个D地址) |
| **M区寄存器** | M200可用 | 用于全局运行互锁信号 |

## 3. ST调用示例

> 详细完整ST代码请参考旧版本文档或DSN文档。
> 核心要点:
> - 4个输入必须正确连接(3个报警码 + 1个复位)
> - 11个输出需要主控读取并写入D/M区
> - 特别注意o_b新报警标志的脉冲特性(仅当周期TRUE)

## 4. D区回写操作指南

### 4.1 主控程序Step10中的回写代码

```st
(*==============================================================================
  主控程序 (PRG_MainControl_DJ2026005_V4.1.0) Step10: 报警D区回写
  ==============================================================================*)

(* 将公共报警FB的输出写入固定D区地址 *)
D400 := fb公共报警.o_w全局报警字;        (* HMI报警灯指示 *)
D402 := fb公共报警.o_i当前报警代码;       (* HMI文本显示 *)
D404 := fb公共报警.o_iMES报警计数;         (* MES累计统计 *)

(* MES报警队列回写 (10 x INT = 20个D地址) *)
D406 := fb公共报警.o_iMES报警队列[0];     (* 最新一条 *)
D408 := fb公共报警.o_iMES报警队列[1];
D410 := fb公共报警.o_iMES报警队列[2];
D412 := fb公共报警.o_iMES报警队列[3];
D414 := fb公共报警.o_iMES报警队列[4];
D416 := fb公共报警.o_iMES报警队列[5];
D418 := fb公共报警.o_iMES报警队列[6];
D420 := fb公共报警.o_iMES报警队列[7];
D422 := fb公共报警.o_iMES报警队列[8];
D424 := fb公共报警.o_iMES报警队列[9];     (* 最旧一条 *)

(* 全局互锁信号写入M区 *)
M200 := fb公共报警.o_b任何报警激活;       (* 运行互锁 *)
```

## 5. MES数据上传触发机制

### 5.1 新报警检测与上传流程

```
新报警检测流程:

每个扫描周期:
  │
  ├─ 读取 o_b新报警标志
  │    │
  │    ├─ FALSE → 无操作, 继续下一周期
  │    │
  │    └─ TRUE → 检测到上升沿!
  │           │
  │           └─ 执行MES上传:
  │               ├─ 读取 o_iMES报警队列[0..9] (D406~D425内容)
  │               ├─ 读取 o_iMES报警计数 (D404)
  │               ├─ 读取 o_i当前报警代码 (D402)
  │               ├─ 组装MES报文(按通信协议格式化)
  │               └─ 通过通信口发送给MES服务器
  │
  下一周期:
  └─ o_b新报警标志 自动恢复为FALSE (单周期脉冲)
      等待下一个新报警事件...
```

### 5.2 MES报文格式建议

```json
{
  "timestamp": "2026-04-25T14:30:00",
  "equipment_id": "DJ-2026-005",
  "alarm_code": 3,
  "alarm_count": 15,
  "recent_alarms": [3, 7, 101, 102, 201, null, null, null, null, null]
}
```

## 6. 调试步骤建议

### 6.1 分阶段调试流程

```
阶段1: 基础连接验证 [预计20分钟]
│  ├─ Step 1.1: 确认3个工站报警码输入连接正确
│  ├─ Step 1.2: 验证复位按钮功能
│  └─ Step 1.3: 监控D400/D402/M200回写正常

阶段2: 单站报警测试 [预计30分钟]
│  ├─ Step 2.1: 仅触发四层输送机报警(如ALM_1)
│  ├─ Step 2.2: 验证D400≠0, D402=1, M200=TRUE
│  ├─ Step 2.3: 触发复位, 验证全部清零
│  └─ Step 2.4: 对取放料机构和送料机构重复测试

阶段3: 多站同时报警测试 [预计30分钟]
│  ├─ Step 3.1: 同时触发多工站报警(如ALM_1 + ALM_102 + ALM_201)
│  ├─ Step 3.2: 验证o_i当前报警代码=1(MIN优先级)
│  ├─ Step 3.3: 验证o_w全局报警字为OR组合结果
│  └─ Step 3.4: 验证各分站状态输出正确

阶段4: MES队列功能测试 [预计30分钟]
│  ├─ Step 4.1: 连续触发不同报警码(至少5种)
│  ├─ Step 4.2: 检查D406~D425队列内容(去重、时序)
│  ├─ Step 4.3: 验证o_b新报警标志脉冲特性
│  ├─ Step 4.4: 测试队列满后覆盖行为(超过10条)
│  └─ Step 4.5: 验证复位后队列清空

阶段5: MES联调测试 [预计60分钟]
│  ├─ Step 5.1: 连接MES服务器
│  ├─ Step 5.2: 测试完整上传流程(触发→脉冲→组装→发送→确认)
│  └─ Step 5.3: 24小时稳定性测试
```

## 7. 常见问题排查指南

### 7.1 问题诊断表 (精选)

| 序号 | 现象 | 可能原因 | 解决方法 | 相关检查点 |
|:----:|:-----|:---------|:---------|:---------|
| 1 | D400始终为0但有报警 | ① 输入未连接<br>② 工站FB未输出非零值<br>③ 本FB未被调用 | 检查3个INT输入是否连接；检查上游FB状态；确认主控Step7调用了本FB | i_i输送机/取放料/送料机构 报警 |
| 2 | D402显示错误的优先级 | ① MIN算法逻辑错误<br>② 某工站报警码超出定义范围 | 检查本FB源代码逻辑；确认报警码在有效范围内(1~7/101~108/201~205) | o_i当前报警代码 vs 三站输入 |
| 3 | MES队列无数据 | ① 未触发过新报警<br>② o_b新报警标志未被检测到<br>③ D区回写代码缺失 | 人为制造报警事件；检查主控脉冲检测逻辑；检查Step10代码 | o_b新报警标志 / D406~D425 |
| 4 | M200未置位但D400≠0 | ① o_b任何报警激活输出异常<br>② M200被其他地方强制清零 | 检查本FB输出逻辑；搜索所有对M200的写操作 | o_b任何报警激活 / M200监控 |
| 5 | 复位后仍有残留报警 | ① 复位信号未到达本FB<br>② 上游工站未清除内部报警 | 检查i_b复位连接；先复位各工站再复位本FB | i_b复位 / 各工站报警码 |

## 9. 使用约束与禁忌

### 9.1 禁止事项

| 序号 | 禁止操作 | 后果 |
|:----:|:--------|:-----|
| 1 | 将多个公共报警FB实例并联使用 | 导致重复计数和队列混乱 |
| 2 | 在主控中直接修改D400/D402等地址 | 与本FB输出冲突，导致数据不一致 |
| 3 | 忽略o_b新报警标志而持续轮询D区 | 浪费CPU资源，且可能漏掉短暂的报警脉冲 |
| 4 | 将报警码输入硬编码为固定值 | 失去真实的报警检测功能 |
