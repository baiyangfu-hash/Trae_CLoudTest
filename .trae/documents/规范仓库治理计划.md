# 规范仓库治理计划 — 专业级文档管理升级

## 问题诊断总览

### 问题一：新旧版本共存（🔴 严重）

| 规范 | 旧版(应处理) | 新版(保留) | 状态 |
|------|-------------|-----------|------|
| 801_PLC命名规范 | `...DEV-V1.0.2.md` | `...DEV-V1.0.3.md` | ⚠️ 旧版残留 |
| 050_模板结构规范 | `...TPL-V1.0.0.md` (05_模板规范目录) | `...TPL-V1.0.1.md` (05_文档模板规范目录) | ⚠️ 旧版残留 + **路径不一致** |
| 004_版本管理规范 | 已被用户删除 | `...DEV-V1.1.1.md` | ✅ 已清理 |
| 810_PLC编程规范 | 原V1.0.0已被覆盖 | `...DEV-V1.0.1.md` | ✅ 已更新 |

**根因**：Subagent执行更新时采用"Write新文件"而非"SearchReplace原地编辑"，导致创建副本而非就地升级。

**额外冗余**：
- `_冲突文件_` 后缀文件 × 3个（`001`/`010`/`031` 各一个）
- `_archive/` 目录下旧版本 × 5个
- **总计约12个冗余文件**

### 问题二：技术栈查找困难（🔴 严重）

**当前目录结构**（纯按过程阶段分类）：
```
00_Obsidian_Base全局规范文件仓库/
├── 01_启动过程/     ← 项目立项 + 基础规范(命名/版本)
├── 02_规划过程/     ← 需求 + 技术规划 + 项目管理
├── 03_执行过程/     ← 代码开发(Python+PLC混放) + 部署 + 测试
├── 04_监控和控制/   ← 变更管理
└── 05_收尾过程/     ← 验收 + 总结
```

**痛点场景**：
- PLC工程师要找PLC相关规范 → 需遍历01(命名) + 02(接口模板) + 03(编程规范) = **3个目录**
- Python工程师找开发规范 → 03/代码开发 下有Python和PLC**混放**
- 新人入门 → 不知道从哪开始看

### 问题三：文档质量缺乏体系化保障（🟡 中等）

**缺失项**：
- ❌ 无"规范发布前检查清单"
- ❌ 无"规范间一致性自动验证"
- ❌ 无正确的版本更新操作SOP（导致问题一）
- ❌ 无"规范废弃/归档标准流程"
- ❌ 部分旧规范自身不符合050模板要求

## 治理方案（三阶段）

### Phase 1: 清理存量 — 版本共存修复（立即执行）

#### Step 1.1: 冗余文件清理
| 操作 | 文件 | 处置方式 |
|------|------|----------|
| 归档 | `801_..._V1.0.2.md` | 移入 `_archive/` |
| 删除 | `050_..._V1.0.0.md` (05_模板规范目录) | 旧路径是错误位置，直接删除 |
| 清理 | 3个 `_冲突文件_` | 移入 `_archive/` |
| 确认 | `_archive/` 现有5个文件 | 保持不动（已是正确归档） |

#### Step 1.2: 路径不一致修复
050规范的旧版在 `05_模板规范/` 目录，新版在 `05_文档模板规范/` 目录。
- 确认 `05_文档模板规范/` 为正确目录
- 删除 `05_模板规范/` 目录下的旧版（Step 1.1已包含）
- 检查是否有其他类似的目录分裂

#### Step 1.3: 建立版本管理SOP
新增一份操作指南（放在 `.trae/documents/` 或规范仓库内）：
```
规范更新标准操作流程(SOP):
1. Read原文件 → 确认内容
2. SearchReplace原地修改 → 不创建新文件
3. 更新版本号（就地修改）
4. 更新变更记录表（追加行）
5. Grep验证无残留旧引用
6. 如需归档旧版 → Git操作或移入 _archive/
```

#### Step 2.2: 利用Obsidian Tag增强查找
为所有规范文件添加YAML frontmatter中的tags：
```yaml
---
tags: [规范/PLC, 规范/命名, 优先级-P0]
---
```
这样在Obsidian中可以按Tag过滤：点击"规范/PLC"标签即可显示所有PLC相关规范。

#### Step 2.3: 规范编号规则增强（可选，Phase 2后期）
当前编号已基本合理（8xx=PLC专用, 2xx=Python, 0xx=通用），但可考虑：
- 在索引文档中正式文档化编号分配规则
- 为未来新规范预留编号段（如9xx=机器人, 7xx=上位机）

### Phase 3: 文档质量保障体系（长期建设）

#### Step 3.1: 新增"规范发布检查清单"
创建 `05_收尾过程/规范发布检查清单_CHK-V1.0.0.md`：
```
## 发布前必查项
- [ ] 文档基础信息完整（6字段）
- [ ] 版本变更记录已更新
- [ ] 编号遵循001/004规范
- [ ] 格式遵循050模板规范
- [ ] 交叉引用链接有效
- [ ] 无旧版本号残留
- [ ] 同类规范间无矛盾
- [ ] Markdown渲染正常
```

#### Step 3.2: 规范自身质量审计
抽查现有规范，标记不符合要求的：
- 缺少基础信息的 → 补充
- 缺少变更记录的 → 补充
- 格式不统一的 → 对齐050标准

#### Step 3.3: 建立"规范评审周期"
建议每季度一次：
- 检查是否有过时规范
- 收集使用反馈
- 执行必要更新
